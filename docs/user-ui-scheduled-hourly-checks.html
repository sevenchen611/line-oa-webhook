每小時檢查程式

每天早上 8 點到晚上 10 點左右，SevenAM 會分段檢查會議、LINE 訊息、任務狀態與權責候選資料。

08:00-22:15 Asia/Taipei Render Cron

檢查項目

時間程式名稱主要內容具體做法
每小時整點 npm run meetings:sync -- --limit 50 掃描新的 SevenAM 會議紀錄，建立會議待辦與專案進度。 讀取會議紀錄資料庫。會議中只要有 checkbox 行動項目，就直接轉入總控任務庫並標示為已確認；會議正文、決議與進度也會用來建立或補充專案進度報表。
每小時 10 分 npm run line:judgements -- --include-outgoing-groups --limit 50 判斷新的 LINE 訊息是否更新既有任務，或是否形成新任務。 先讀同一 LINE 群組前後文，再比對活躍總控任務。若訊息是回覆、補資料、完成、卡住、改時間、改負責人，就更新既有任務；只有無法被既有任務吸收且有真實行動時，才新增事件層級任務。
每小時 15 分 npm run responsibility:sync 更新權責判斷用的候選 LINE 群組與候選負責人。 依照權責定義表的「第一層：總控專案」列出候選 LINE 群組，再依已選群組列出候選成員，讓任務後續能更精準找到主要負責人。
本機完整執行 npm run assistant:hourly 一次跑完整 hourly 維護流程。 依序更新 LINE 群組選項、同步會議待辦、判斷 LINE 訊息、更新權責候選資料。這是本機人工檢查或校準時使用的完整流程。

判斷原則

核心問題這則新訊息是在更新既有任務，還是一個真正需要新增的新事件？
優先做法先更新既有任務，再考慮新增任務，避免一則 LINE 訊息變成一筆任務。
會建立任務新交辦、新外部承諾、新財務/人資/法務/稅務/客訴風險、新決策、新待追蹤結果。
不建立任務背景資訊、寒暄、重複訊息、測試文字、純知識分享、已被前後文完成的問題、Seven Jr. 查詢或打開任務的操作指令。
更新證據任務狀態改變時，會把來源對話、發話者、新訊息摘要、前後文與更新原因寫回任務內文或來源欄位。
每小時檢查的目的不是製造更多任務，而是讓總控任務庫保持乾淨：已完成的要更新，重複的要合併，無行動的要排除，真正新的事件才新增。
